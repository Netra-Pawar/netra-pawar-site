# Exercise 1: Audience Analysis & Workflow Mapping

## My methodology

I always start a documentation redesign the same way: with the readers, not the content.

The instinct at most companies is to begin with what exists — audit the pages, reorganise the
nav, rewrite the worst offenders. That approach optimises for the content you already have.
It does not ask whether that content is the content anyone needs.

My process has three stages:

1. **Audience analysis** — who comes to the docs, from where, carrying what context, and
   in what emotional state. Not role titles. Real readers with real constraints.
2. **Workflow mapping** — for each audience segment, what are the specific tasks that
   bring them to the docs? What question are they trying to answer? What does success
   look like, and how quickly do they need it?
3. **Information architecture** — only after the first two stages are locked. The IA is
   the structure that serves the workflows of the audiences. It is downstream of both.

This document covers stages 1 and 2. The IA redesign (stage 3) is a separate deliverable
that depends on what we establish here.

## Stage 1: Audience analysis

### The axis that matters: not "who are you" but "what are you trying to do right now"

Most documentation teams segment their audience by role title: developer, admin, enterprise
user. Mistral's current docs do this too — the three-product split (Vibe, Studio, Admin)
implicitly maps to "individual user," "developer," and "IT administrator."

This is a reasonable starting point, but it misses a critical dimension: **journey stage**.
A developer evaluating whether to use Mistral at all and a developer debugging a Workflow
retry failure at 11pm on a Friday need completely different content, even though they share
the title "developer." The Evaluator needs breadth: a model comparison, a pricing page, a
quickstart that proves the API works. The debugger needs depth and speed: an error code
reference, a troubleshooting guide, a code sample they can paste without reading three
paragraphs of context first.

Segmenting by journey stage surfaces readers that a role-based taxonomy misses entirely.
It also reveals the emotional spectrum of documentation usage — a dimension that most
documentation strategies ignore.


### The dual-consumer reality: humans and machines

There is a second axis that most documentation redesigns in 2025 missed and that no
documentation strategy in 2026 can afford to ignore: **docs are no longer consumed only
by humans.**

As of mid-2026, more than 30% of the increase in API demand is coming from AI tools —
coding agents, RAG pipelines, and autonomous systems that read documentation
programmatically to figure out how to call APIs, choose models, and handle errors.

This means Mistral's docs have two classes of consumer:

**Human readers** who navigate via the sidebar, scan headings, and read prose.
They benefit from narrative structure, conceptual framing, and visual cues like
code tabs and callout boxes.

**Machine readers** — AI coding agents (Cursor, Claude Code, GitHub Copilot,
and Mistral's own Vibe Code), RAG pipelines, and customer-built support bots that
ingest docs to answer questions. They benefit from:

- **Semantic structure over visual structure.** A heading hierarchy that conveys
  meaning, not just visual weight. Frontmatter metadata (title, description, tags,
  audience, capability) that machines can parse without reading the prose.
- **Token-efficient content.** Dense, direct language. Fewer filler paragraphs.
  Code samples that are self-contained (all imports, all context) so the agent
  does not need to infer what was omitted.
- **Consistent naming.** If the model string is `mistral-medium-latest` on one
  page and `Mistral Medium 3.5` on another, a human infers they are the same
  thing. An agent may not. Every reference to a model, endpoint, parameter, or
  product should use the canonical name, and there should be exactly one
  canonical name.
- **Explicit relationships.** "See also" links are invisible to an agent that
  fetches a single page. Cross-references need to be semantic (tagged in
  frontmatter as `related: [/studio/agents, /models/mistral-medium-3-5]`),
  not just hyperlinked in prose.

The practical implication: every content decision I make from here on asks two questions.
Does this help a human developer find what they need? And does this help an AI agent
extract the right answer on first retrieval?

Where those two goals conflict (rare, but real — narrative context helps humans but
adds noise for machines), the answer is structured content with progressive disclosure:
the machine-readable version is the semantic skeleton (frontmatter, headings, code blocks,
parameter tables), and the human-readable version layers narrative prose around it.

This is not a future consideration. Mistral's own Vibe Code is an AI coding agent that
reads documentation to generate code. If Vibe Code cannot reliably extract the correct
model string, endpoint, or parameter format from Mistral's own docs, that is a product
bug, not a docs bug. The documentation is part of the product surface.


### The nine reader segments

The table below segments Mistral's documentation audience along four axes: how they arrive
at the docs (entry point), what stage of their journey they are in, which product surface
they touch, and how deeply they read code.

I have deliberately separated segments that role-based taxonomies collapse together.
The Competitive Evaluator and the Self-Serve Builder are both "developers," but they
need different content, different depth, and different speed. The Enterprise Buyer and
the Solutions Engineer both care about deployment, but the Buyer reads prose and the
SE reads architecture diagrams.

| #  | Segment | Entry point | Journey stage | Product surface | Technical depth | Emotional state |
|----|---------|-------------|---------------|-----------------|-----------------|-----------------|
| 1  | Competitive Evaluator | Google search, Hacker News, comparison article, competitor docs | Evaluating — forming an opinion in ≤90 minutes | Models, pricing, quickstart | Reads code but will not write it yet | Sceptical, time-pressured, comparing tabs |
| 2  | Self-Serve Builder | Console → docs link, quickstart search, SDK README | Building — first integration | Studio / API | Writes code, copies samples | Optimistic but impatient — wants a working call in &lt;5 min |
| 3  | Vibe Work User | In-app help link, product onboarding, Google "how to use Mistral" | Using (not building) — personal productivity | Vibe Work only | Does not read code | Curious, possibly confused by Vibe/Le Chat rename |
| 4  | Vibe Code Developer | GitHub README, CLI `--help`, word of mouth, Vibe Code docs link | Building — agent-driven development | Vibe Code | Writes code, uses CLI | Experimental, wants to see what it can do |
| 5  | Production Operator | Search on error code, bookmarked API reference, Stack Overflow | Troubleshooting or scaling — something is broken or needs tuning | API reference, observability, error codes | Deep technical — reads response headers | Frustrated, time-pressured, often late at night |
| 6  | IT / Security Admin | Invited by an internal champion, procurement checklist | Governing — setting up org, SSO, billing, policies | Admin only | Semi-technical — reads config, not code | Methodical, checkbox-driven |
| 7  | Enterprise Buyer / Compliance Reviewer | Link shared by Mistral sales, own Google search for "Mistral GDPR" | Evaluating at org level — not a technical evaluation | Deploy, security, compliance (mostly absent today) | Non-technical — reads prose, ctrl-F for keywords | Risk-averse, needs confidence signals |
| 8  | Solutions Engineer / Partner | Internal wiki, live customer call, pre-sales prep | Spans all stages, acting on behalf of a customer | Everything, especially deployment architecture | Technical — reads and demos | Under pressure to have the answer now, on a call |
| 9  | AI Agent / Machine Reader | Programmatic fetch, RAG ingestion, Vibe Code context retrieval | Continuous — no journey, just retrieval | API reference, model cards, parameter docs | Parses structure, not prose | N/A — but quality of retrieval directly affects end-user experience |


### Edge cases the table does not capture

The nine segments cover the primary audiences. But documentation also serves readers in
states that cut across all segments:

**The frustrated user.** Someone whose API call just returned a `429` or a `422` and who
is not in a learning mindset — they want the answer, not the explanation. Every piece of
orientation content between them and the fix (a conceptual introduction, a "what you'll
learn" preamble, a table of contents) is friction. This reader needs: error code → cause →
fix, in that order, on one page, with zero preamble. The current docs do not have a
dedicated error reference page.

**The migrator.** Someone moving from OpenAI or Anthropic to Mistral. They do not need to
learn what an API is. They need a mapping: "If you were using `gpt-4o`, the equivalent
Mistral model is `mistral-medium-latest`. Here is how the request format differs." This
reader does not exist in Mistral's current docs. A migration guide would be high-value,
low-effort content that directly serves acquisition.

**The returning user after a product rename.** Le Chat became Vibe in mid-2026. A user who
bookmarked a Le Chat docs page or who learned the product under its old name now
encounters a vocabulary mismatch. The Vibe overview page carries an info box ("Le Chat
is now Vibe"), but the sidebar on some pages still lists "Le Chat" as a nav category.
This reader needs consistent naming, working redirects, and zero cognitive overhead
figuring out whether their bookmarks still apply.

**The Mistral customer-support agent (human).** A CS team member who gets a customer
question and needs to either answer it directly or find a docs link to share. If they
cannot find the answer in the docs, they write a custom response — which means Mistral
is paying for documentation it has not written. This reader needs the docs to be the
single source of truth, not a secondary source behind the blog or internal wiki.

## Stage 2: Workflow mapping

For each of the nine segments, I am mapping the specific workflows (tasks) that bring
them to the docs, the question they are trying to answer, what success looks like, and
the speed at which they need the answer. This is the foundation the IA redesign will
be built on: every section in the nav should map to at least one high-priority workflow.
Any section that does not serve a workflow is either missing its audience or is dead
weight.


### Segment 1: Competitive Evaluator

The Evaluator is forming an opinion. They are not yet a customer.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **Model comparison** | "Which model should I use for my use case, and how does it compare to GPT-4o / Claude Sonnet?" | A decision table: use case → model → price → context → license. One page. | &lt; 3 minutes |
| **Pricing check** | "How much will this cost me at 10M tokens/month?" | A pricing page with per-model token costs, a calculator or formula, and batch discounts. | &lt; 2 minutes |
| **Proof of life** | "Does the API actually work? Can I get a response in &lt;5 minutes?" | A quickstart that runs on first paste. API key → SDK install → response printed. | &lt; 5 minutes |
| **Differentiation scan** | "Why Mistral over OpenAI/Anthropic? What is the unique selling point?" | A page that states the differentiation directly: open-weight models, EU sovereignty, on-prem deployment, Apache 2.0 licensing. | &lt; 2 minutes |
| **Trust verification** | "Is this a real company? Who uses it?" | Customer logos, case study links, SOC 2 / GDPR compliance statements. | &lt; 1 minute (visual scan) |

**Current docs gap:** The Model Selection Guide now renders comparison content, but it is still not a true use-case-first decision guide. The competitor migration guide exists, but is buried under Developers / Resources rather than surfaced during evaluation. Pricing content lives primarily on mistral.ai/pricing, not in the docs — which means the Evaluator has to leave the docs to complete the decision. The quickstarts do exist and are genuinely good.


### Segment 2: Self-Serve Builder

The Builder has decided to use Mistral. They are writing production code.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **First API call** | "How do I install the SDK and get a response?" | Working code, printed response, &lt;5 minutes. | &lt; 5 minutes |
| **Tool use / function calling** | "How do I let the model call my functions?" | A code sample with tool definition, model response, and function execution loop. | &lt; 15 minutes |
| **RAG setup** | "How do I upload documents and query them?" | Working RAG pipeline: upload → index → query → cited response. | &lt; 20 minutes |
| **Workflow orchestration** | "How do I build a durable, multi-step pipeline?" | A working Workflow with activities, a worker, and a trigger. | &lt; 20 minutes |
| **Model version pinning** | "Should I use `mistral-large-latest` or `mistral-large-2512`? What happens when a new version ships?" | A clear explanation of latest vs. dated model strings, deprecation policy, and how to pin for stability. | &lt; 3 minutes |
| **Structured output** | "How do I get JSON output reliably?" | A code sample with `response_format`, a schema definition, and error handling for malformed responses. | &lt; 10 minutes |
| **Streaming** | "How do I stream responses to my UI?" | A code sample with SSE handling, token-by-token display, and connection error handling. | &lt; 10 minutes |
| **Going to production** | "What do I need to worry about before going live?" | A checklist or guide covering: rate limits, retry strategy, model pinning, cost estimation, security best practices. | &lt; 15 minutes (read once) |

**Current docs gap:** The quickstarts are solid. Tool use, RAG, and Workflows have dedicated
pages. But there is no "going to production" guide, no model versioning explainer, no
structured output guide, and no streaming guide beyond the API reference. The gap between
"quickstart complete" and "production-ready" is where Mistral loses Builders who have
already decided to try the platform.


### Segment 3: Vibe Work User

This reader uses Mistral through the Vibe web/mobile app. They do not write code.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **First task** | "How do I get Vibe to do something useful?" | A walkthrough: open Vibe → describe a task → see it execute → review the output. | &lt; 5 minutes |
| **Skill creation** | "How do I save a repeatable process?" | A guide: what a Skill is, how to create one, how to reuse it. | &lt; 10 minutes |
| **Connector setup** | "How do I connect my Google Drive / Slack / CRM?" | Step-by-step with screenshots: authenticate → grant permissions → verify connection. | &lt; 5 minutes |
| **Le Chat → Vibe migration** | "I was using Le Chat. What changed? Where are my conversations?" | A mapping: old feature → new location. Reassurance that history is preserved. | &lt; 2 minutes |
| **Data privacy** | "Is my data being used to train models?" | A direct, linkable answer. Yes/no, with the relevant terms of service referenced. | &lt; 1 minute |

**Current docs gap:** The Vibe section is the best-organised part of the docs. The Work
quickstarts exist and are time-estimated. The gap is the Le Chat migration — there is
an info box but no dedicated mapping page. And the data privacy question, which is the
number-one concern for enterprise Vibe Work users, does not have a standalone answer page in
the docs.


### Segment 4: Vibe Code Developer

This reader uses Mistral through the CLI or VS Code extension for AI-assisted coding.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **CLI installation** | "How do I install Vibe Code and get it running?" | A command sequence: install → configure API key → first prompt → code generated. | &lt; 5 minutes |
| **Project scaffolding** | "Can Vibe Code generate a project from a description?" | A walkthrough: describe → generate → review files → run. | &lt; 10 minutes |
| **VS Code setup** | "How do I use Vibe Code in my editor?" | Extension install → configure → use in-editor. | &lt; 5 minutes |
| **Remote sessions** | "How do I run Vibe Code in a cloud sandbox?" | Setup guide for Vibe Code Web. | &lt; 10 minutes |
| **Context management** | "How does Vibe Code decide which files to read? Can I scope it?" | Explanation of context window, file inclusion/exclusion, project-level config. | &lt; 5 minutes |

**Current docs gap:** This section exists and is reasonably well-structured. The main
gap is context management — a common question for users of any AI coding agent, and one
where Mistral could differentiate against Cursor and Claude Code.


### Segment 5: Production Operator

This reader has a working integration and something has gone wrong, or they need to scale.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **Error diagnosis** | "What does this error code mean and how do I fix it?" | Error code → cause → fix. No preamble. One page with all error codes. | &lt; 30 seconds |
| **Rate limit understanding** | "What are my rate limits? How do I request more?" | A table: plan → model → RPM/TPM limits. A link to request increases. | &lt; 1 minute |
| **Retry strategy** | "How should I handle 429s in production?" | A code sample with exponential backoff, jitter, and circuit breaker pattern. | &lt; 5 minutes |
| **Cost investigation** | "Why is my bill higher than expected?" | Explanation of token counting, batch vs. real-time pricing, cost per model. | &lt; 5 minutes |
| **Latency debugging** | "Why are my responses slow?" | Checklist: model choice, prompt length, streaming vs. sync, region, concurrency. | &lt; 5 minutes |
| **Deprecation response** | "A model I depend on is being deprecated. What do I do?" | Migration guide: old model → new model → code changes required → timeline. | &lt; 10 minutes |
| **Observability setup** | "How do I monitor my API usage and catch problems early?" | Guide to Studio observability features, or integration with external monitoring. | &lt; 15 minutes |

**Current docs gap:** This is the most underserved segment. There is no error code
reference page. There is no rate limit table. There is no retry strategy guide. There
is no deprecation/migration guide. The Production Operator's workflows are almost entirely
unserved by the current docs — which means every one of these questions becomes a support
ticket. This is the single highest-ROI documentation gap.


### Segment 6: IT / Security Admin

This reader is setting up Mistral for their organisation.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **Org setup** | "How do I create an org, invite members, and set roles?" | Step-by-step guide. | &lt; 15 minutes |
| **SSO configuration** | "How do I set up SAML SSO?" | A procedure with screenshots and IdP-specific instructions (Okta, Azure AD). | &lt; 30 minutes |
| **Workspace isolation** | "How do I isolate teams so they cannot see each other's work?" | Explanation of workspace model, permissions, API key scoping. | &lt; 10 minutes |
| **Billing setup** | "How do I set up billing, set budgets, and get invoices?" | Procedure: payment method → budget alerts → invoice access. | &lt; 10 minutes |
| **Audit and compliance** | "What audit logs are available? Can I export them?" | Explanation of log coverage, retention, export format. | &lt; 5 minutes |

**Current docs gap:** The Admin section is competent and well-scoped. The gaps are
audit/compliance depth (what exactly is logged, and is it enough for SOC 2 evidence?)
and IdP-specific SSO instructions (the current guide is generic SAML).


### Segment 7: Enterprise Buyer / Compliance Reviewer

This reader is not technical. They are evaluating Mistral for organisational adoption.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **Data residency** | "Does our data leave our infrastructure?" | A direct answer: on-prem deployment available, EU jurisdiction, no data used for training (if applicable). Linkable. | &lt; 1 minute |
| **Regulatory compliance** | "Is Mistral SOC 2 certified? GDPR compliant? EU AI Act ready?" | A compliance page listing certifications, with dates, scope, and downloadable attestation letters. | &lt; 2 minutes |
| **Security architecture** | "What is the security model? Encryption at rest? In transit?" | A security whitepaper or overview page. Prose, not code. | &lt; 10 minutes |
| **Deployment options** | "Can we run this entirely on-prem? What are the options?" | A deployment comparison: cloud, hybrid, self-hosted. What each includes and excludes. | &lt; 5 minutes |
| **Data processing agreement** | "Where is the DPA?" | A link to the DPA, ideally with a summary of key terms. | &lt; 30 seconds |

**Current docs gap:** This is the second most underserved segment after the Production
Operator. There is no compliance page, no security overview, no DPA link in the docs,
and no deployment comparison written for a non-technical reader. The blog post announcing
OCR 4 does a better job explaining data sovereignty than anything in the docs. This means
every enterprise deal requires a Solutions Engineer to manually assemble compliance answers
from scattered sources — slowing deal velocity and creating inconsistency across accounts.


### Segment 8: Solutions Engineer / Partner

This reader is technical but is acting on behalf of a customer, often in real time.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **Architecture demo** | "Show me the deployment topology for an on-prem enterprise setup." | An architecture diagram with components, data flows, and network boundaries. Shareable, screenshot-able, no login required. | &lt; 1 minute (they need to find it during a call) |
| **Feature-to-customer mapping** | "The customer needs OCR + RAG in a VPC. Which products, which models, what pricing?" | A solution blueprint: product → model → deployment → pricing → code sample. | &lt; 5 minutes |
| **Customer objection handling** | "The customer says they cannot use a French AI company because of US export controls. What is our position?" | A compliance/sovereignty page with accurate, quotable statements about jurisdiction, data handling, and certifications. | &lt; 2 minutes |
| **Pre-sales technical validation** | "Can I run a proof-of-concept with the customer using their data without leaving their infra?" | A self-hosted deployment quickstart with clear hardware requirements and networking configuration. | &lt; 30 minutes |
| **Demo preparation** | "I need to demo Workflows and OCR 4 tomorrow. What is the fastest path to a working demo?" | Links to the best quickstarts + a "demo recipe" that chains them into a coherent story. | &lt; 15 minutes |

**Current docs gap:** No architecture diagrams. No solution blueprints. No compliance
page with quotable statements. The deployment content exists but is model-focused, not
architecture-focused. The SE's most common workflows are almost entirely unsupported,
which means SEs build their own slide decks from blog posts — creating inconsistency
across deals and wasting expensive headcount on content that should be canonical.


### Segment 9: AI Agent / Machine Reader

This is not a human persona but a consumption pattern that must be designed for explicitly.

| Workflow | Core question | Success = | Speed requirement |
|----------|---------------|-----------|-------------------|
| **Model selection** | "Given a task description, which model string should I use?" | Structured data: model name → API string → capabilities → pricing → context window. Parseable without prose. | Milliseconds (retrieval) |
| **API call construction** | "What is the exact request format for a chat completion with tool use?" | A complete, self-contained code sample with all parameters, all imports, and expected response shape. | Milliseconds |
| **Parameter validation** | "Is `reasoning_effort` a valid parameter for this model? What are the allowed values?" | A parameter table with: name, type, allowed values, default, models that support it. | Milliseconds |
| **Error interpretation** | "The API returned `422`. What does the error body mean?" | Error code → structured cause → resolution steps. Machine-readable (JSON or table). | Milliseconds |
| **Deprecation awareness** | "Is the model string I am using still valid? Is there a successor?" | Model status (active/deprecated) and successor model string in a machine-readable format. | Milliseconds |

**Current docs gap:** The API reference exists and is auto-generated from an OpenAPI spec,
which is good for machine readability. But the model cards, capability pages, and guides
are prose-heavy and lack structured frontmatter that an agent could parse. There is no
machine-readable model registry (a JSON endpoint or structured page listing all active
models with their strings, statuses, and capabilities). Mistral's own Vibe Code is a
consumer of this content — if it cannot reliably extract the correct model string or
parameter format, that is a product-level issue.

## Cross-cutting observations

### The emotional spectrum of documentation

One thing this workflow mapping surfaces is that documentation serves readers across
a wide emotional spectrum:

- **Curious and exploratory** — the Evaluator, the Vibe Code Developer trying a new feature.
  These readers tolerate and even welcome conceptual framing, narrative context, and
  "what you'll learn" preambles.
- **Focused and building** — the Self-Serve Builder following a quickstart. These readers
  want clear steps, runnable code, and "what's next" links. They are patient but will
  abandon if a code sample fails.
- **Frustrated and urgent** — the Production Operator at 11pm, the SE on a customer call
  who cannot find the architecture diagram. These readers need the answer in the first
  paragraph, ideally in the first sentence. Every piece of orientation content is friction.
  Every conceptual introduction is a barrier.
- **Cautious and risk-averse** — the Enterprise Buyer, the Compliance Reviewer.
  These readers need confidence signals: certifications, dates, scopes, downloadable
  attestation letters. They are not reading for understanding — they are reading to
  check boxes.

A documentation architecture that serves all four states needs to separate orientation
content (conceptual introductions, "why this matters") from reference content (the
answer itself). The current docs intermingle these, which means the frustrated operator
has to scroll past "Workflows let you run multi-step AI pipelines that survive crashes"
to find the error code table that does not yet exist.


### The workflows that generate the most support tickets

Based on the gaps identified above, these are the workflows most likely to generate
support tickets today because the docs do not serve them:

1. **Error diagnosis** (Production Operator) — error content exists, but is under-surfaced from the production path
2. **Data residency / compliance** (Enterprise Buyer) — no single compliance page
3. **Architecture / deployment** (Solutions Engineer) — no architecture diagrams
4. **Rate limits** (Production Operator) — rate-limit content needs to be surfaced from the production path
5. **Model selection** (Evaluator, AI Agent) — Model Selection Guide exists, but needs a clearer decision workflow
6. **Model migration / deprecation** (Production Operator) — no deprecation guide
7. **Going to production** (Builder) — no production readiness guide

These seven gaps are where the documentation redesign should start. They represent the
highest-ROI content investments because each one directly reduces support load, accelerates
deals, or prevents churn.


### What this means for the IA redesign

The workflow mapping directly informs the information architecture in three ways:

**1. The Production Operator needs a first-class home.**
Today, there is no section of the docs designed for someone who already has a working
integration. The nav goes: Getting Started → (build things) → API Reference. The
operator's workflows (error diagnosis, rate limits, retry strategy, cost investigation)
have no home. I would add a "Production" or "Operate" section between Studio and Admin.

**2. Enterprise content needs to be findable without a login.**
The Enterprise Buyer and the Solutions Engineer both need content they can share with
customers who do not have Mistral accounts. This means: no auth-gated pages for
compliance, security, and deployment architecture content. It also means these pages
need to be discoverable via Google, not just via the sidebar nav.

**3. AI agents need structured data, not just structured prose.**
The API reference (auto-generated from OpenAPI) is already machine-readable. But model
cards, capability guides, and error documentation are not. Adding frontmatter metadata
(model strings, capabilities, status, audience tags) to every page would make the entire
docs site function as a knowledge base for both human and machine readers — and would
make Mistral's own Vibe Code better at generating correct code.
