# Gap Analysis: Pages ↔ Audience Workflows (v2)

## What changed from v1

The first draft of this analysis was built on an under-verified content inventory. A
dedicated fact-checking pass caught three real errors, all in the same direction — content
that exists was classified as missing:

1. **Migration guide from competitors (was M-09, "missing")** — a full migration guide
   already exists at `/resources/migration-guides`, covering OpenAI and self-hosted Llama
   migration paths with working code. It is not missing. It is misplaced — buried under
   Developers > Resources instead of surfaced to Evaluators during their decision window.
2. **Error code reference (was M-03, "missing")** — the error glossary at
   `/resources/error-glossary` already documents every HTTP status code, the full error
   JSON schema, and a working exponential-backoff retry sample. It is not missing. It is
   under-surfaced — a Production Operator debugging a `429` has no reason to look under
   "Developers > Resources" for it.
3. **Retry / error handling strategy (was M-07, "missing")** — same page, same correction.
   The retry code sample already exists on the error glossary page.

This matters beyond just getting three rows right. It changes the *type* of fix each one
needs. "Missing content" means someone has to write new documentation from scratch —
research the topic, draft it, review it, ship it. "Misplaced content" means the content
already exists and the fix is architectural: add a cross-link, duplicate a summary, move
a nav entry. The second kind of fix is roughly an order of magnitude cheaper and faster
to ship. Recommending "write a new production guide" when the real problem is "nobody
surfaced the guide that already exists" would have been a wasted recommendation — and
worse, if implemented, would have created a duplicate, competing page instead of fixing
the actual problem (findability).

The rest of the original analysis held up under re-verification and is carried forward
with minor corrections noted inline.

### What this changes about the recommended structural moves

The three big structural conclusions from v1 still hold, but with sharper framing now
that the "missing vs. misplaced" split is accurate:

**1. The "Deploy" top-level section is still the single biggest gap**, and it's now more
precisely scoped: it needs to house genuinely new content (M-05 compliance page, M-06
architecture diagrams, M-13 security overview) *and* absorb two pages that already exist
elsewhere but belong there (Cloud deployments, Self-deployment).

**2. The "Production" section under Studio is now a smaller lift than v1 suggested.**
Two of its three anchor pages — error handling and rate limits — already exist and are
well-built. The section mostly needs to *aggregate and cross-link* existing content,
plus author the one piece that's genuinely missing: a unified model-pinning and
cost-estimation guide (M-02, M-10, M-14).

**3. The Le Chat → Vibe migration cleanup should be presented cautiously**, because a final review found the developer sidebar issue no longer reproducing in fetched text. The durable point is still valid: product renames require redirects, migration notes, and stale-label audits.
the sidebar problem isn't "old section vs. new section," it's page-by-page inconsistency
within the same folder — which argues for a systematic redirect audit across every single
page in Getting Started, not just the obviously-old ones.

---

*This is the corrected foundation for the IA redesign. The proposed architecture should
distinguish, in its own presentation, between sections that require new writing and
sections that require re-surfacing existing work — because the second category is
something a hiring panel can reasonably expect to ship in week one, and conflating the
two would understate how much of this problem is genuinely solvable fast.*
