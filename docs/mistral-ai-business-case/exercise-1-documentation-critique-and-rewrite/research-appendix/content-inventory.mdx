# docs.mistral.ai — Complete Content Inventory (v2)

## Verification methodology

This is a full rewrite of the initial inventory, not a patch. A dedicated verification
pass was run after the first draft, and it caught real errors — some claims were
under-evidenced, some URLs were wrong, some content marked "missing" in downstream
analysis turned out to exist. That pass is folded into this version. The standard applied:

- **Confirmed** — verified by direct fetch of the live page, in most cases more than once.
- **Confirmed via search index** — the page wasn't directly fetched but its content was
  corroborated through indexed search snippets showing real body text (not just meta tags).
- **Structure known, URL unconfirmed** — the page's existence is implied by a sidebar
  listing (its title appears as a nav item) but the exact slug was not independently
  verified. These are marked explicitly rather than presented as fact. This replaces the
  previous version's practice of writing "(inferred)" next to a guessed URL — a guessed
  URL is not useful information, so where I could not confirm a slug, I now say so plainly
  and omit the guess.

Any claim in this document without one of these markers should be treated as unconfirmed
and re-checked before being relied on.



> **Freshness note — July 1, 2026:** A final review found that two previously high-severity observations appear to have changed on the live site: `/models/model-selection-guide` now renders body content, and the developer quickstart pages now show the newer Vibe-oriented sidebar consistently in fetched text. I would not present the older "empty page" or "mixed sidebar" claims as current defects without screenshots/time-stamped evidence. The more durable critique is that model-selection content still behaves like a catalogue rather than a decision workflow, and tabbed code examples duplicate in extracted text, which matters for AI-reader legibility.

Page count: **~125–135 pages** across 6 top-level sections, plus the API reference.

**On the Model Selection Guide:** final review found the page now rendering useful body content under the visible heading "Compare Models." I would not present the earlier empty-page observation as a current defect without screenshots or timestamped evidence. The stronger current critique is that the page contains rich comparison data, but it still does not behave like a use-case-first decision guide.

**Model cards** — confirmed real, individual pages at `/models/model-cards/{slug}`.
Each card renders an interactive widget (confirmed via fetch) showing speed, performance,
modalities, context window, and price. Confirmed pricing for six models, direct from the
widget:

| Model | Context | Input price | Output price | Versioned API string |
|-------|---------|-------------|---------------|----------------------|
| Mistral Medium 3.5 | 256k | $1.5/M tokens | $7.5/M tokens | `mistral-medium-3-5` |
| Mistral Small 4 | 256k | $0.15/M tokens | $0.6/M tokens | `mistral-small-2603` |
| Mistral Large 3 | 256k | $0.5/M tokens | $1.5/M tokens | `mistral-large-2512` |
| Ministral 3 14B | 256k | $0.2/M tokens | $0.2/M tokens | `ministral-14b-2512` |
| OCR 4 | — | $4/1,000 pages | $5/1,000 annotated pages | `mistral-ocr-4-0` |

Worth noting because it's counterintuitive and easy to get wrong by assumption: **Mistral
Large 3 is priced lower than Mistral Medium 3.5**, despite the tier name suggesting
otherwise. Large 3 is the open-weight (Apache 2.0), self-hostable flagship; Medium 3.5 is
the managed, agentic-optimized model — that's presumably the reason for the inversion, but
that's my inference, not something the docs state directly.

Separately, `-latest` aliases (`mistral-large-latest`, `mistral-medium-latest`, etc.) are
confirmed in use throughout quickstart code samples and the migration guide — both the
versioned string and the `-latest` alias are real and valid, they just serve different
purposes (pinning vs. always-current).

**Legacy/Deprecated models table** — recounted precisely from two direct fetches of
`/models/overview`: **45 rows.** The original inventory said "35+," which understated it.

One row is a genuine content anomaly, not a miscount: **Devstral 2** appears simultaneously
in the "Featured Models" section at the top of the page (marked as a current, active
model) **and** in the Legacy/Deprecated table lower on the same page, both showing the
identical version `25.12`. Confirmed via two separate fetches, so this is a real
inconsistency on Mistral's live site, not a fetch artifact — worth flagging as its own
finding, since it means a reader scanning the Legacy table for guidance could reasonably
conclude a currently-recommended model is deprecated.

### Content check: Error glossary (corrected characterization)

The original inventory listed this page correctly by URL but did not read its actual
content. Direct fetch confirms it contains:

- Full HTTP status code reference (400, 401, 403, 404, 422, 429, 500, 502, 503, 504) with plain-language causes
- The complete error response JSON schema (`message`, `type`, `param`, `code` fields)
- **A working exponential-backoff retry code sample** in Python, plus a note that the official SDKs implement this automatically

This page is substantially more complete than it was given credit for downstream (see
Gap Analysis v2).

### Content check: Known limitations (new detail)

Direct fetch confirms this page documents: context window limits and the resulting `400`
error, rate limit enforcement mechanics (RPS and TPM enforced independently, per
organization, batch exempted), file retention (30 days), max batch file size (512 MB) —
and, notably: **"The Mistral API is served from EU data centers by default. Some models
may not be available in all regions."** This is a real, if minimal, fragment of the data
residency information that the Enterprise Buyer persona needs — it exists, but not as a
dedicated, citable compliance page. This matters for the Gap Analysis (see M-05 correction).

### Content check: Migration guides (major correction)

The original inventory correctly listed this URL but the downstream Gap Analysis
mischaracterized its content as being about *model deprecation* and separately claimed
a competitor-migration guide was *missing entirely*. Both were wrong. Direct fetch
confirms this page:

- Explicitly covers **"From OpenAI"** and **"From self-hosted Llama"** as two dedicated migration paths
- States plainly that the Mistral Chat Completions API mirrors the OpenAI request structure
- Provides working code samples showing the exact three changes needed (client import, base URL, model name) to migrate from OpenAI's SDK
- Notes tokenizer differences for Llama migrations
- Confirms OpenAI-compatible clients (LangChain, LlamaIndex) work by changing only base URL and model name

This is real, well-built content serving exactly the workflow the Gap Analysis (v1)
claimed was missing. See Gap Analysis v2 for the corrected classification.
