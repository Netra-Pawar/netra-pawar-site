# Quickstart: Embed Content and Search by Meaning

**Embed three documentation chunks and a user question, then find the best semantic match.**

Time to complete: ~5 minutes.

**Prerequisites:** A [Mistral API key](https://console.mistral.ai/api-keys/) set as `MISTRAL_API_KEY` in your environment. Python 3.8+ with `mistralai` and `numpy` installed, or Node.js 18+ with `@mistralai/mistralai`.

## Embed and query

You have three documentation chunks and a user question. The goal is to find which chunk best matches the question's meaning.

### Python

```python
import os
import numpy as np
from mistralai import Mistral

client = Mistral(api_key=os.environ["MISTRAL_API_KEY"])

# Your documentation chunks
documents = [
    "Credential renewal procedure: create a new API key, update your "
    "application configuration, verify traffic, then revoke the old key.",
    
    "Rate limits protect the API from excessive traffic. If you exceed "
    "your rate limit, requests return HTTP 429.",
    
    "Use webhooks to receive real-time event notifications when "
    "resources change in your account."
]

# User's question
query = "How do I rotate my API key?"

# Step 1: Embed the documents
doc_response = client.embeddings.create(
    model="mistral-embed",
    inputs=documents
)

# Step 2: Embed the query
query_response = client.embeddings.create(
    model="mistral-embed",
    inputs=[query]
)

# Step 3: Compare — find the most similar document
# mistral-embed returns normalized vectors, so dot product = cosine similarity.
query_vector = np.array(query_response.data[0].embedding)
doc_vectors = [np.array(d.embedding) for d in doc_response.data]

similarities = [np.dot(query_vector, doc_vec) for doc_vec in doc_vectors]

# In production, batch all inputs in a single call and use a vector store
# for efficient similarity search instead of computing manually.
best_index = int(np.argmax(similarities))
print(f"Best match (score: {similarities[best_index]:.4f}):")
print(documents[best_index])
```

<summary>TypeScript version</summary>

```typescript
import { Mistral } from "@mistralai/mistralai";

const client = new Mistral({ apiKey: process.env.MISTRAL_API_KEY });

const documents = [
  "Credential renewal procedure: create a new API key, update your " +
  "application configuration, verify traffic, then revoke the old key.",

  "Rate limits protect the API from excessive traffic. If you exceed " +
  "your rate limit, requests return HTTP 429.",

  "Use webhooks to receive real-time event notifications when " +
  "resources change in your account."
];

const query = "How do I rotate my API key?";

const docResponse = await client.embeddings.create({
  model: "mistral-embed",
  inputs: documents,
});

const queryResponse = await client.embeddings.create({
  model: "mistral-embed",
  inputs: [query],
});

// Dot product (mistral-embed vectors are normalized)
const queryVec = queryResponse.data[0].embedding;

const similarities = docResponse.data.map((doc) => {
  const docVec = doc.embedding;
  return queryVec.reduce((sum, v, i) => sum + v * docVec[i], 0);
});

const bestIndex = similarities.indexOf(Math.max(...similarities));
console.log(`Best match (score: ${similarities[bestIndex].toFixed(4)}):`);
console.log(documents[bestIndex]);
```

## Expected output

Your output will look similar to:

```
Best match (score: 0.8XXX):
Credential renewal procedure: create a new API key, update your
application configuration, verify traffic, then revoke the old key.
```

The exact score varies, but the credential renewal document should rank highest by a clear margin — the embedding model understood that "rotate my API key" and "credential renewal procedure" mean the same thing, even though they share almost no keywords.

## What should you embed?

You rarely embed an entire document as one input. Long documents contain multiple topics, and a single embedding for all of them loses specificity.

Instead, **split documents into chunks** — meaningful sections, paragraphs, or logical units — and embed each chunk separately.

| What to embed | Why it works |
|---|---|
| A paragraph or section about one topic | Focused meaning, good retrieval |
| A single FAQ entry (question + answer) | Natural query-answer pair |
| A function docstring or code comment | Matches natural-language code questions |

| What to avoid as a single input | Why it causes problems |
|---|---|
| An entire 20-page document | Too many topics dilute meaning |
| A table extracted as raw text | Structure lost, rows become noise |
| A heading with no body text | Not enough content to represent meaning |

Getting chunk boundaries right matters more than most developers expect. If the answer to a user's question is split across two chunks, neither chunk will be a strong match on its own.

:::note
In a production application, you would store document vectors in a vector database (such as FAISS, Pinecone, Weaviate, or Qdrant) and use its built-in similarity search instead of computing manually. See the [RAG Quickstart](/studio-api/knowledge-rag/rag_quickstart) for a working FAISS example, or use [Libraries](/studio-api/knowledge-rag/libraries) for fully managed retrieval.
:::
