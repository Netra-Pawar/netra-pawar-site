# Vector Embeddings in Production Retrieval Systems

**Design, evaluate, and debug semantic retrieval for RAG and AI assistants.**

{/* import RetrievalDiagnostic from './RetrievalDiagnostic' */}

New to embeddings? Start with [Build Semantic Search with Mistral Embeddings](/studio-api/knowledge-rag/build-semantic-search).

---

## Embeddings as a retrieval primitive

A vector embedding is a dense numerical representation of an input, optimized so that semantically related inputs are close under a similarity function. In a RAG system, embeddings serve a specific and limited role: **they select candidate context for the model to use.** They do not generate answers, verify correctness, or enforce access control.

This distinction matters in production. When a RAG application returns an incorrect answer, the root cause is either a retrieval failure (the right content was not in the candidate set) or a generation failure (the model misinterpreted the context). Conflating the two makes debugging impossible.

---

## Retrieval architecture

Embeddings appear in two paths:

```
INDEXING PATH
source → parse → chunk → embed → store (vector + metadata)

QUERY PATH
query → embed → ANN search → filter → rerank → context assembly → generation
```

**Indexing** converts source documents into searchable vectors. Each step is a potential failure point: a parser that drops table structure, a chunking boundary that splits a procedure across two chunks, or a stale index that still contains deprecated content.

**Query processing** converts the user's input into the same vector space and retrieves the nearest neighbors. The retrieved set is then narrowed by metadata filters, optionally reranked, and assembled into a prompt.

### Mistral embedding models

| Model | Dimensions | Context | Best for | Pricing |
|---|---|---|---|---|
| `mistral-embed` | 1,024 | — | Natural-language documents, knowledge bases, support content | Standard |
| `codestral-embed` | Up to 3,072 (configurable via `output_dimension`) | 8,192 tokens | Source code, repository search, code analytics | $0.15/M tokens |

`codestral-embed` supports Matryoshka-style dimensionality reduction — dimensions are ordered by information density, so truncating to 1,024 or 256 dimensions trades precision for speed without re-embedding.

:::note SDK vs REST parameter
The current `mistralai` Python and TypeScript SDKs use **`inputs`** (plural). The REST endpoint accepts **`input`** (singular). Mixing these up is a common source of silent failures. See [Text Embeddings](/studio-api/knowledge-rag/embeddings/text_embeddings) for verified examples in both formats.
:::

## What affects retrieval quality

Retrieval quality is not solely a property of the embedding model. It is a joint function of the model, the content, and the pipeline. The following factors interact:

### Content and ingestion

**Chunk size and boundaries.** Smaller chunks (200–500 tokens) produce more specific embeddings but risk splitting answers across chunks. Larger chunks (500–2,000 tokens) preserve more context but dilute semantic specificity. There is no universal optimum — it depends on document structure and query patterns.

For code embeddings, Mistral recommends chunks of approximately 3,000 characters with 1,000 characters of overlap. For text, the [Libraries](/studio-api/knowledge-rag/libraries) feature accepts chunk sizes between 256 and 32,768 characters; the [Search Toolkit](/studio-api/knowledge-rag/search-toolkit) exposes `MarkdownTextSplitter`, `TokenTextSplitter`, and `CharacterTextSplitter` with configurable `chunk_size` and `chunk_overlap`.

**Structure preservation.** Tables, FAQ sections, collapsible panels, and code blocks carry structure that plain-text extraction often destroys. A table row like `| mistral-embed | 1024 | text |` extracted as flat text loses its column relationships. If your source content is heavily structured, evaluate extraction quality before tuning the embedding model.

**Metadata quality.** Metadata (product version, content type, language, publication date, access tier) enables post-retrieval filtering. Missing or inconsistent metadata is a common cause of both false positives (wrong version surfaces) and false negatives (correct content filtered out).

**Source freshness.** If the indexing pipeline runs nightly but documentation updates deploy continuously, there is a window where the vector store contains stale representations. Track index freshness lag as an operational metric.

### Query and retrieval

**Query specificity.** Vague queries like `"help with my account"` produce generic embeddings that match many chunks weakly. Specific queries like `"how to rotate an API key without downtime"` produce embeddings with strong directional signal. Query rewriting (expanding abbreviations, adding context) can improve retrieval for terse inputs — the Search Toolkit provides `LLMQueryRewriter` and `LLMQueryExtension` for this.

**Near-duplicates.** Multiple versions of the same document, localized variants, or content syndicated across products create clusters of near-identical vectors. The "correct" result may be outranked by a close duplicate. Deduplication during ingestion or reranking during retrieval can address this.

**Domain vocabulary.** General-purpose embedding models may underperform on specialized terminology (internal product names, acronyms, domain jargon). Evaluate retrieval accuracy on queries that use your domain's vocabulary before committing to a model.

## Semantic similarity is not correctness

This is the most consequential distinction in production retrieval:

> Embedding search retrieves semantically **similar** candidates. It does not guarantee that a candidate is **correct, current, authorized, complete, or canonical.**

Concrete failure scenarios:

| Scenario | What happens | Why similarity fails |
|---|---|---|
| Deprecated documentation | A retired API guide is semantically identical to its replacement | Similarity has no concept of currency |
| Near-duplicate outranking | The v1 version of a page outranks the v2 version | Both are semantically close; ranking is arbitrary without metadata |
| Partial chunk | A chunk contains the concept but omits the critical warning or prerequisite | The chunk boundary was poorly placed |
| Permission-excluded content | The canonical source is in a restricted corpus that the user's query doesn't reach | Retrieval respects filter boundaries, not semantic relevance |
| Structurally complex content | The answer is in a table, but the table was extracted as flat text and embedded poorly | Extraction quality, not embedding quality, is the root cause |

The mitigation is not a better embedding model. It is metadata filtering (version, status, permissions), reranking (promoting authoritative sources), and content pipeline hygiene (correct extraction, deduplication, freshness tracking).

## Text, code, and multimodal considerations

**Text embeddings** (`mistral-embed`) are appropriate for natural-language documents: knowledge bases, support articles, product documentation, policies.

**Code embeddings** (`codestral-embed`) are optimized for source code: repository search, code analytics, finding functions by description. They support configurable output dimensions (default 1,536, max 3,072) and quantized output types (`float`, `int8`, `binary`, `ubinary`) for storage-constrained deployments.

**Multimodal content** (PDFs with diagrams, mixed text-image documents) does not have a dedicated Mistral embedding model. The current approach is extraction-then-embedding: use [Document AI](/studio-api/document-ai) or the Search Toolkit's `MistralOCRExtractor` to extract text from visual content, then embed the extracted text with `mistral-embed`. This works well for text-heavy documents but loses visual information that is not captured in the extracted text.

Do not mix embedding models within a single vector index. A query embedded with `mistral-embed` cannot meaningfully be compared against document vectors produced by `codestral-embed` — the vector spaces are different.

## Failure modes and diagnostics

When retrieval returns wrong or missing results, the failure almost always originates in the pipeline, not the embedding model. Diagnose by stage:

| Pipeline stage | Symptom | Diagnostic |
|---|---|---|
| **Ingestion** | Content never appears in results | Was the source document crawled, uploaded, or connected? Check ingestion logs for errors or omissions. |
| **Parsing** | Content appears but with garbled structure | Inspect the raw extracted text. Did tables retain column associations? Did HTML/Markdown conversion preserve headings? |
| **Chunking** | Relevant content appears in results but doesn't fully answer the question | Inspect the chunk that was retrieved. Is the answer split across the boundary? Does the chunk include enough surrounding context? |
| **Embedding** | Semantically relevant content ranks low | Is the content type appropriate for the model? Is domain terminology represented? Try embedding the query and the target chunk directly and inspect their similarity score. |
| **Indexing** | Results are stale or missing recent updates | When was the index last rebuilt? Is the indexing pipeline running on schedule? Compare the document's last-modified timestamp against the index timestamp. |
| **Retrieval** | Wrong chunks outrank correct ones | Inspect the top-k results. Are near-duplicates consuming slots? Is `k` too small? |
| **Filtering** | Known-good content doesn't appear | Review active metadata filters. Are version, product, or permission filters excluding the target? |

:::tip Isolate retrieval from generation
Always evaluate retrieval quality independently. Embed the query, inspect the top-k chunks, and verify that the correct content appears in the candidate set **before** examining the generated answer. If the right chunk isn't retrieved, prompt engineering will not fix the response.
:::

## Evaluation metrics

### Retrieval metrics

Measure how well the embedding + retrieval pipeline surfaces relevant content:

| Metric | What it measures | When to use |
|---|---|---|
| **Recall@k** | Fraction of relevant documents in the top-k results | Primary metric — does the correct content appear in the candidate set? |
| **Precision@k** | Fraction of top-k results that are relevant | High precision = less noise for the model to filter through |
| **MRR** (Mean Reciprocal Rank) | Average position of the first correct result | Matters when the model primarily uses the top-1 or top-2 chunks |
| **nDCG** (Normalized Discounted Cumulative Gain) | Ranking quality, weighted by position | Use when the ordering within top-k matters, not just presence |
| **No-result rate** | Fraction of queries that return zero results | Indicates coverage gaps in the corpus or overly aggressive filtering |
| **Stale-result rate** | Fraction of results where the source has been updated since indexing | Tracks index freshness |
| **Retrieval latency (p50/p95/p99)** | Time from query to candidate set | Embedding inference + ANN search + filter time |

### End-to-end metrics

These measure the complete RAG pipeline, not just retrieval:

| Metric | What it measures |
|---|---|
| **Answer acceptance rate** | How often users accept the generated answer (via thumbs up, follow-up behavior, or absence of reformulation) |
| **Grounded-answer rate** | How often the model's answer is traceable to a retrieved chunk |
| **Escalation rate** | How often users escalate to human support after receiving an answer |
| **Source correctness** | How often the cited source is the canonical, current version |

> **Measure retrieval before generation.** If you only measure end-to-end quality, you cannot distinguish a retrieval failure (right chunk never surfaced) from a generation failure (model ignored or misinterpreted the chunk). Evaluate both stages independently before optimizing either.

The [Search Toolkit](/studio-api/knowledge-rag/search-toolkit) supports evaluation with recall, precision, MRR, and nDCG against custom test sets.

## Production considerations

### Latency budget

Embedding a single query with `mistral-embed` via the API typically completes in under 200ms. Add ANN search time (usually sub-50ms for moderate-scale indexes), filter/rerank time, and context assembly. For interactive applications, target total retrieval latency under 500ms (p95) — but benchmark in your own environment, since latency varies with payload size, region, and concurrent load.

### Cost at scale

Embedding large corpora is a one-time cost at indexing time plus incremental cost for updates. Query-time embedding cost scales linearly with query volume. For high-throughput applications, consider:

- Caching frequently seen queries with the Search Toolkit's `SemanticCache`.
- Using `codestral-embed` with reduced `output_dimension` to lower storage costs.
- Batch embedding via the [Batch API](/studio-api/batch-processing) at 50% discount.

### Hybrid search

Pure vector search misses exact-match cases (error codes, product IDs, config keys). Pure keyword search misses semantic equivalents. Combine both:

- The Search Toolkit supports `VectorRetriever` + `BM25Retriever` with `RRFRanker` (reciprocal rank fusion) for hybrid search.
- [Libraries](/studio-api/knowledge-rag/libraries) provides managed hybrid retrieval without pipeline construction.

### Permission-aware retrieval

Enterprise deployments typically require filtering results by user permissions (team, role, data classification). This is a metadata filter applied **after** ANN search, which means:

- The vector index must include permission metadata at ingestion time.
- Post-filter top-k may return fewer than k results if most candidates are filtered out. Over-retrieve (e.g., search for 50, filter to 10) to compensate.
- Permission changes require metadata updates in the index, not re-embedding.

### Observability

Log enough to diagnose failures after the fact:

- The raw query.
- The embedded query vector (or a hash for lookup).
- The top-k chunk IDs and similarity scores.
- Which filters were applied and how many candidates they removed.
- Whether reranking changed the ordering.
- Total retrieval latency broken down by stage.

## Architecture decision: API, Libraries, or Search Toolkit?

Mistral offers three levels of abstraction for embedding-based retrieval:

| Approach | When to use | What you manage |
|---|---|---|
| **Embeddings API** (`client.embeddings.create`) | Building a custom pipeline with your own vector store, chunking, and retrieval logic | Everything: ingestion, chunking, storage, search, filtering, reranking |
| **[Libraries](/studio-api/knowledge-rag/libraries)** | Need managed RAG without building infrastructure — upload documents, get retrieval | Nothing: Mistral handles ingest, embedding, storage, and search |
| **[Search Toolkit](/studio-api/knowledge-rag/search-toolkit)** | Building a production pipeline with control over chunking, hybrid search, reranking, and evaluation | Pipeline logic: you configure components but use Mistral's framework for orchestration |

Choose Libraries if speed-to-production matters more than pipeline control. Choose the Search Toolkit if you need to tune retrieval quality, run evaluations, or integrate with existing infrastructure. Use the raw Embeddings API only if you need full control or are integrating into a non-Python stack.
