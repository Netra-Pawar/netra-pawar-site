# Build Semantic Search with Mistral Embeddings

**Help users find the right content — even when they ask in different words.**

{/* import SemanticSearchDemo from './SemanticSearchDemo' */}

---

## The problem: keywords miss what users mean

Imagine you're building a documentation assistant. A user asks:

> *"How do I rotate my API key?"*

Your documentation contains the answer, but the page is titled **"Credential renewal procedure"** and the text reads:

> *Create a new API key, update your application configuration, verify traffic, then revoke the old key.*

A keyword search for `rotate API key` will not match `credential renewal`. The words are different. But the **meaning** is the same.

This is the problem embeddings solve. Instead of matching words, you match meaning.

## When to use embeddings

Embeddings are useful when users search by **meaning** rather than exact terms:

- A **documentation assistant** where users describe problems in their own words.
- A **support bot** that matches incoming tickets to existing solutions.
- An **internal knowledge search** across policies, runbooks, or product specs.
- A **code search tool** that finds relevant functions from natural-language descriptions.
- Any **RAG application** that retrieves context before generating an answer.

### When embeddings alone are not enough

Embeddings find content that is **semantically similar**. They do not handle:

- **Exact identifiers** — order numbers, error codes, product SKUs. Use keyword search or filters.
- **Dates and time ranges** — "tickets from last week." Use metadata filters.
- **Access control** — which documents a user is allowed to see. Use permission filters.
- **Structured lookups** — pricing tables, config values. Use database queries.

Most production systems combine embeddings with keyword search and metadata filters. Embeddings handle the meaning; filters handle the facts.

:::note
Embeddings retrieve **candidate content** for a model to use. They do not generate answers. If you're building a RAG application, embeddings are the retrieval step — the model generates the response from the retrieved content.
:::

## Where embeddings fit in a RAG workflow

If you're building a RAG application, embeddings appear at two points in the pipeline:

```
┌─────────────────── INDEXING (done once per document) ──────────────────┐
│                                                                        │
│   Your documents → Split into chunks → Embed each chunk → Store vectors│
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘

┌─────────────────── QUERY (done for every user question) ──────────────┐
│                                                                        │
│   User question → Embed the question → Find similar chunks →          │
│   Send chunks to the model → Model generates an answer                 │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

The key insight: **the same embedding model encodes both your documents and the user's question** into the same space, so "rotate my API key" lands near "credential renewal procedure" — even though the words are different.

---

## What is a vector embedding?

A vector embedding is a list of numbers that represents the meaning of a piece of text. The Mistral `mistral-embed` model produces a list of 1,024 numbers for each input.

You don't need to interpret these numbers directly. What matters is the relationship between them:

- Texts with **similar meanings** produce vectors that are **close together**.
- Texts with **different meanings** produce vectors that are **far apart**.

Think of it as coordinates on a map: "rotate API key" and "credential renewal" end up in the same neighborhood, while "rate limit configuration" is in a different part of the map entirely.

## Try it: embed a query and find the best match

Here's a minimal example. You have three documentation chunks and a user question. The goal is to find which chunk is most relevant.

### Python

```python
import os
import numpy as np
from mistralai import Mistral

client = Mistral(api_key=os.environ["MISTRAL_API_KEY"])

# Your documentation chunks
documents = [
    "Credential renewal procedure: create a new API key, update your "
    "application configuration, verify traffic, then revoke the old key.",
    
    "Rate limits protect the API from excessive traffic. If you exceed "
    "your rate limit, requests return HTTP 429.",
    
    "Use webhooks to receive real-time event notifications when "
    "resources change in your account."
]

# User's question
query = "How do I rotate my API key?"

# Step 1: Embed the documents
doc_response = client.embeddings.create(
    model="mistral-embed",
    inputs=documents
)

# Step 2: Embed the query
query_response = client.embeddings.create(
    model="mistral-embed",
    inputs=[query]
)

# Step 3: Compare — find the most similar document
query_vector = np.array(query_response.data[0].embedding)
doc_vectors = [np.array(d.embedding) for d in doc_response.data]

# Cosine similarity (higher = more similar)
similarities = [
    np.dot(query_vector, doc_vec) / 
    (np.linalg.norm(query_vector) * np.linalg.norm(doc_vec))
    for doc_vec in doc_vectors
]

# Find the best match
best_index = int(np.argmax(similarities))
print(f"Best match (score: {similarities[best_index]:.4f}):")
print(documents[best_index])
```

### TypeScript

```typescript
import { Mistral } from "@mistralai/mistralai";

const client = new Mistral({ apiKey: process.env.MISTRAL_API_KEY });

const documents = [
  "Credential renewal procedure: create a new API key, update your " +
  "application configuration, verify traffic, then revoke the old key.",

  "Rate limits protect the API from excessive traffic. If you exceed " +
  "your rate limit, requests return HTTP 429.",

  "Use webhooks to receive real-time event notifications when " +
  "resources change in your account."
];

const query = "How do I rotate my API key?";

// Step 1: Embed documents
const docResponse = await client.embeddings.create({
  model: "mistral-embed",
  inputs: documents,
});

// Step 2: Embed query
const queryResponse = await client.embeddings.create({
  model: "mistral-embed",
  inputs: [query],
});

// Step 3: Compare with cosine similarity
const queryVec = queryResponse.data[0].embedding;

const similarities = docResponse.data.map((doc) => {
  const docVec = doc.embedding;
  const dot = queryVec.reduce((sum, v, i) => sum + v * docVec[i], 0);
  const normQ = Math.sqrt(queryVec.reduce((s, v) => s + v * v, 0));
  const normD = Math.sqrt(docVec.reduce((s, v) => s + v * v, 0));
  return dot / (normQ * normD);
});

const bestIndex = similarities.indexOf(Math.max(...similarities));
console.log(`Best match (score: ${similarities[bestIndex].toFixed(4)}):`);
console.log(documents[bestIndex]);
```

### Expected output

```
Best match (score: 0.8432):
Credential renewal procedure: create a new API key, update your
application configuration, verify traffic, then revoke the old key.
```

The query `"How do I rotate my API key?"` matched the credential renewal document — not because they share keywords, but because the embedding model understood they mean the same thing.

:::note
In a production application, you would store document vectors in a vector database (such as FAISS, Pinecone, Weaviate, or Qdrant) and use its built-in similarity search instead of computing similarity manually. The example above shows the mechanics; a vector store handles the scaling. See the [RAG Quickstart](/studio-api/knowledge-rag/rag_quickstart) for a working FAISS example, or use [Libraries](/studio-api/knowledge-rag/libraries) for fully managed retrieval.
:::

## What should you embed?

You rarely embed an entire document as one input. Long documents contain multiple topics, and a single embedding for all of them loses specificity.

Instead, **split documents into chunks** — meaningful sections, paragraphs, or logical units — and embed each chunk separately.

| What to embed | Why it works well |
|---|---|
| A paragraph or section about one topic | Focused meaning, good retrieval |
| A single FAQ entry (question + answer) | Natural query-answer pair |
| A function docstring or code comment | Matches natural-language code questions |

| What to avoid embedding as a single input | Why it causes problems |
|---|---|
| An entire 20-page document | Too many topics in one vector — diluted meaning |
| A table extracted as raw text | Structure is lost, rows become noise |
| A heading with no body text | Not enough content to represent meaning |

Getting the chunk boundaries right matters more than most developers expect. If the answer to a user's question is split across two chunks, neither chunk will be a strong match on its own.

## Common mistakes

**1. Assuming embeddings generate answers.**
Embeddings retrieve content. The language model generates the answer. If your RAG application gives a wrong answer, the problem could be in retrieval (wrong chunk surfaced) or in generation (model misinterpreted the chunk). Isolate which step failed before debugging.

**2. Embedding entire documents instead of chunks.**
A single vector for a long document averages out the meaning. A question about API key rotation might match a document titled "Security Best Practices" but miss the specific paragraph about credential renewal.

**3. Using embeddings for exact-match lookups.**
If the user asks for error code `ERR_AUTH_042`, embeddings will find semantically similar content about authentication errors — not the specific error code entry. Combine with keyword search or metadata filters.

**4. Forgetting that new content isn't searchable until it's embedded.**
If you publish new documentation but don't re-run your embedding pipeline, users won't find it. Embeddings are computed at indexing time, not at query time.

**5. Ignoring source structure during extraction.**
If your documents contain tables, collapsible sections, or embedded diagrams, the text extraction step may lose that structure. A table row like `| mistral-embed | 1024 | text |` extracted as plain text may not embed well. Check how your content looks after extraction.

## Troubleshooting: "Why didn't my assistant find the right answer?"

When your retrieval system misses relevant content, the cause is rarely the embedding model itself. Work through this checklist:

| Check | Question to ask |
|---|---|
| **Was the content indexed?** | Is the document in your vector store? Was it added after the last indexing run? |
| **Was it chunked well?** | Is the answer split across two chunks? Is the relevant paragraph inside a chunk that also contains unrelated content? |
| **Was structure preserved?** | Did tables, FAQ widgets, or collapsible sections lose their structure during text extraction? |
| **Is the index fresh?** | Was the document updated since it was last embedded? Does the old version still appear? |
| **Is the query too vague?** | Does `"help with my account"` have enough meaning for the model to identify the right topic? |
| **Did filters exclude it?** | If you apply metadata filters (product, version, language), could they be removing the correct result? |
| **Is a near-duplicate outranking it?** | Do you have multiple similar documents (e.g., v1 and v2 of the same guide)? The wrong version might score higher. |

:::tip
Test retrieval separately from generation. Embed your query, inspect the top 5 returned chunks, and confirm the right content appears before looking at the generated answer. If the right chunk isn't retrieved, no amount of prompt engineering will fix the response.
:::
