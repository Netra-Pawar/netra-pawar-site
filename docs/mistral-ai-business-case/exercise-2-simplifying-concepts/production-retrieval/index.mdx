# Vector Embeddings in Production Retrieval Systems

**Design, evaluate, and debug semantic retrieval for RAG and AI assistants.**

New to embeddings? Start with [Build Semantic Search with Mistral Embeddings](../build-semantic-search/).

## Embeddings as a retrieval primitive

A vector embedding is a dense numerical representation of an input, optimized so that semantically related inputs are close under a similarity function. In a RAG system, embeddings serve a specific and limited role: **they select candidate context for the model to use.** They do not generate answers, verify correctness, or enforce access control.

This distinction matters in production. When a RAG application returns an incorrect answer, the root cause is either a retrieval failure (the right content was not in the candidate set) or a generation failure (the model misinterpreted the context). Conflating the two makes debugging impossible.

## Retrieval architecture

Embeddings appear in two paths:

```
INDEXING PATH
source → parse → chunk → embed → store (vector + metadata)

QUERY PATH
query → embed → ANN search → filter → rerank → context assembly → generation
```

**Indexing** converts source documents into searchable vectors. Each step is a potential failure point: a parser that drops table structure, a chunking boundary that splits a procedure across two chunks, or a stale index that still contains deprecated content.

**Query processing** converts the user's input into the same vector space and retrieves the nearest neighbors. The retrieved set is then narrowed by metadata filters, optionally reranked, and assembled into a prompt.

### Mistral embedding models

| Model | Dimensions | Context | Best for | Pricing |
|---|---|---|---|---|
| `mistral-embed` | 1,024 | 8,192 tokens | Natural-language documents, knowledge bases, support content | $0.10/M tokens |
| `codestral-embed` | Up to 3,072 (configurable via `output_dimension`) | 8,192 tokens | Source code, repository search, code analytics | $0.15/M tokens |

`codestral-embed` supports Matryoshka-style dimensionality reduction — dimensions are ordered by information density, so truncating to 1,024 or 256 dimensions trades precision for speed without re-embedding.

:::note SDK vs REST parameter
The current `mistralai` Python and TypeScript SDKs use **`inputs`** (plural). The REST endpoint accepts **`input`** (singular). Mixing these up is a common source of silent failures. See [Text Embeddings](/studio-api/knowledge-rag/embeddings/text_embeddings) for verified examples in both formats.
:::

## Semantic similarity is not correctness

This is the most consequential distinction in production retrieval:

> Embedding search retrieves semantically **similar** candidates. It does not guarantee that a candidate is **correct, current, authorized, complete, or canonical.**

| Scenario | What happens | Why similarity fails |
|---|---|---|
| Deprecated documentation | A retired API guide is semantically identical to its replacement | Similarity has no concept of currency |
| Near-duplicate outranking | The v1 version of a page outranks the v2 version | Both are semantically close; ranking is arbitrary without metadata |
| Partial chunk | A chunk contains the concept but omits the critical warning or prerequisite | The chunk boundary was poorly placed |
| Permission-excluded content | The canonical source is in a restricted corpus that the user's query doesn't reach | Retrieval respects filter boundaries, not semantic relevance |
| Structurally complex content | The answer is in a table, but the table was extracted as flat text and embedded poorly | Extraction quality, not embedding quality, is the root cause |

The mitigation is not a better embedding model. It is metadata filtering (version, status, permissions), reranking (promoting authoritative sources), and content pipeline hygiene (correct extraction, deduplication, freshness tracking).
