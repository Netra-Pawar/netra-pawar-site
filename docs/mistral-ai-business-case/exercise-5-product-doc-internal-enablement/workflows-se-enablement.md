## What Workflows is

Workflows is Mistral's orchestration platform for production AI. It lets customers chain model calls, tool use, external APIs, and human approvals into multi-step pipelines that survive crashes, restarts, and transient failures without losing state.

**Your one-liner for the room:**

> "Workflows is the layer between your models and your business process. It handles durability, retries, observability, and human-in-the-loop so your engineering team writes business logic, not infrastructure plumbing."

**When to reach for Workflows (and when not to):**

Workflows is the right fit when the customer's AI use case involves more than a single prompt-response call. If they're building a pipeline that chains multiple model calls, needs to pause for human approval, hits external APIs, or has to run for longer than a single request-response cycle, that's Workflows territory.

If the customer is making a single LLM call with no orchestration, plain SDK calls are sufficient. Don't oversell.

## How it works

### The two building blocks

There are only two primitives a customer's team needs to learn:

**Workflows** are the orchestration layer. They coordinate steps, make decisions, hold state, and wait for events. Workflow code must be deterministic (the platform replays it from event history on restart — more on that below). A workflow contains no I/O, no API calls, no side effects.

**Activities** are the units of real work. Every operation that touches the outside world — LLM calls, HTTP requests, database writes, file reads — goes in an activity. Activities don't need to be deterministic and are automatically retried with configurable backoff when they fail.

The rule of thumb for your prospect: "If it talks to the outside world, it's an activity. If it decides what happens next, it's a workflow."

Here's what the hello-world workflow looks like. This is the actual code generated by the CLI scaffold (not a hypothetical):

```python
# src/workflows/hello.py — generated by `uvx mistralai-workflows-cli setup`

from pydantic import BaseModel
import mistralai.workflows as workflows

class HelloInput(BaseModel):
    name: str = "World"

@workflows.activity()
async def greet(name: str) -> str:
    """A simple activity that returns a greeting."""
    return f"Hello, {name}! Welcome to Mistral Workflows."

@workflows.workflow.define(
    name="hello-world",
    workflow_display_name="Hello World",
    workflow_description="A minimal hello-world workflow.",
)
class HelloWorkflow:
    @workflows.workflow.entrypoint
    async def run(self, input: HelloInput) -> str:
        return await greet(input.name)
```

Point out the signal-to-noise ratio: the entire workflow is 15 lines of business logic. No retry logic, no state management, no queue setup. The platform handles all of that.

### The hybrid architecture

This diagram is the single most important visual in an enterprise security conversation. Draw this on a whiteboard when the CISO asks "where does our data go?"

```
┌──────────────────────────────┐       ┌──────────────────────────────┐
│      MISTRAL  (Studio)       │       │    CUSTOMER ENVIRONMENT      │
│                              │       │                              │
│  Orchestrator                │◄─────►│  Workers                     │
│  (state, history, dispatch)  │       │  (your code, your data)      │
│                              │       │                              │
│  REST API + Studio UI        │       │  Laptop (dev) or             │
│  Le Chat (Vibe Work)         │       │  Kubernetes / VMs (prod)     │
└──────────────────────────────┘       └──────────────────────────────┘
        Workers connect OUTBOUND only.
        Mistral never initiates connections into the customer's network.
```

Three things to stress:

1. **Data residency.** The customer's workflow code and business data run inside their environment. Mistral hosts the orchestrator (state machine, task dispatch, event history) only.
2. **Encryption option.** The SDK can encrypt payloads before they leave the worker. Mistral stores ciphertext only.
3. **On-prem option.** For enterprise customers, the full Studio environment can also run in the customer's private cloud or on-premises.

### How workflows get triggered

This matters for stakeholder conversations. An engineer writes the workflow; a compliance analyst can run it without touching code.

| Surface | Who uses it | How it works |
|---|---|---|
| REST API | Developers, CI/CD pipelines | `POST /v1/workflows/{name}/execute` from any client, any language |
| Studio UI | Developers, ops | Input forms auto-generated from the workflow's Python type signature |
| Le Chat (Vibe Work) | Business users | Workflows appear as assistants anyone in the org can invoke in conversation |

## Demo: Cargo release with human-in-the-loop

This is the example to reach for in enterprise demos. It ships with the SDK scaffold itself (not a hypothetical), it's based on what CMA-CGM actually runs with Workflows, and it demonstrates every key primitive in one flow: OCR extraction, LLM structured output, sub-workflows, conditional branching, human-in-the-loop pause, and certificate generation.

### The story to tell

> "A shipping document arrives. Workflows OCRs the PDF, calls Mistral to classify the cargo, branches on the classification. If it's dangerous goods, a sub-workflow runs additional validation. If anomalies are found, the workflow pauses and waits for a human reviewer — for hours or days, with no compute consumed and no polling. When the reviewer approves in Studio or Le Chat, it resumes exactly where it left off and generates the release certificate. Every step is logged and auditable."

### The code walkthrough

The cargo release example lives in three files. Walk through them in this order during a demo:

**File 1 — `models.py` (Pydantic schemas for structured LLM output):**

```python
class CargoClassification(BaseModel):
    cargo_type: Literal["general", "dangerous_goods", "perishable"]
    reasoning: str = Field(description="Why the LLM chose this classification")
```

What to highlight: the LLM's output is constrained to an enum via `Literal`. No parsing, no regex, no hoping the model returns valid JSON. The SDK validates the response against the schema automatically.

**File 2 — `activities.py` (every piece of real work):**

```python
@workflows.activity(
    name="ocr_extract_shipping_doc",
    retry_policy_max_attempts=3,           # OCR APIs are flaky — retry transient failures
    start_to_close_timeout=timedelta(minutes=2),
)
async def ocr_extract_shipping_doc(document_uri: str) -> str:
    response = await workflows_mistralai.mistralai_ocr(
        params=OCRRequest(
            model="mistral-ocr-latest",
            document=DocumentURLChunk(document_url=document_uri),
        )
    )
    return "\n\n".join(page.markdown for page in response.pages)
```

What to highlight: retry policy and timeout are configured with decorators, not try/except boilerplate. If the OCR call hits a transient 500, the platform retries with exponential backoff automatically.

**File 3 — `workflow.py` (the orchestration — pure control flow, zero I/O):**

```python
@workflows.workflow.define(
    name="cargo-release-compliance",
    execution_timeout=timedelta(hours=24),  # HITL review can take hours
)
class CargoReleaseWorkflow(workflows.InteractiveWorkflow):

    @workflows.workflow.entrypoint
    async def run(self, params: CargoReleaseInput) -> CargoReleaseResult:

        # Step 1: OCR the shipping document
        extracted_text = await ocr_extract_shipping_doc(params.document_uri)

        # Step 2: Classify cargo type (structured LLM output)
        classification = await classify_cargo_type(extracted_text)

        # Step 3: If dangerous goods → run sub-workflow for extra validation
        dg_result = None
        if classification.cargo_type == "dangerous_goods":
            dg_result = await workflows.workflow.execute_workflow(
                DangerousGoodsValidationWorkflow, ...
            )

        # Step 4: Customs compliance check
        compliance = await validate_customs_compliance(...)

        # Step 5: If anomaly detected → pause for human reviewer
        if anomaly_detected:
            confirmation = await self.wait_for_input(
                workflows_mistralai.AcceptDeclineConfirmation(
                    description=f"Shipment {params.shipment_id}: {anomaly_summary}",
                    accept_label="Approve release",
                    decline_label="Block & escalate",
                )
            )
            if not workflows_mistralai.is_accepted(confirmation):
                return CargoReleaseResult(status="blocked", ...)

        # Step 6: Generate release certificate
        certificate = await generate_release_certificate(...)
        return CargoReleaseResult(status="released", ...)
```

### Four things to highlight during the demo

1. **`wait_for_input()` is one line of code.** The workflow durably suspends — no compute, no polling, no heartbeats. It resumes the instant the reviewer submits a decision in Studio or Le Chat. Compare this to building a queue + state machine + notification system + approval UI.

2. **Sub-workflows isolate failure.** `DangerousGoodsValidationWorkflow` has its own execution timeline in Studio. If its activities fail exhaustively, only the sub-workflow is marked failed. The parent decides how to proceed.

3. **Everything is observable.** Open Studio and show the live execution timeline — every activity completion, signal received, and error is a recorded event. This is the audit trail a compliance or risk team needs.

4. **Crash recovery is invisible.** If the worker restarts between Step 2 and Step 5, the platform replays the workflow from event history and resumes from the last completed step. The OCR doesn't re-run, the classification doesn't re-run. The customer's team writes zero recovery code.

## Technical Q&A

### Limits and constraints

| Constraint | Value | What to tell the customer |
|---|---|---|
| Python version | 3.12+ | Standard modern Python. No exotic runtime. |
| SDK version | `mistralai-workflows >= 3.0.0` | Install with `uv add "mistralai-workflows[mistralai]"` |
| Input/output per execution | 2 MB | For larger payloads, use payload offloading to the customer's own S3, GCS, or Azure Blob Storage. Mistral stores only a reference. |
| Execution history | 51,200 events or 50 MB | For very long-running workflows, use `continue_as_new()` to start a fresh history while carrying forward state. |
| Inter-activity deadline | 2 seconds | Workflow code between activity calls must complete within 2 seconds. All heavy computation goes in activities. |
| Default execution timeout | 1 hour | Configurable up to days/weeks: `execution_timeout=timedelta(days=7)` |

### The determinism question

Customers always ask: "What constraints does this put on our code?"

Workflow code must be deterministic because the platform replays it from event history when a worker restarts. In practice, three substitutions:

- `workflow.now()` instead of `datetime.now()`
- `workflow.uuid4()` instead of `uuid.uuid4()`
- `workflow.random()` instead of `random.random()`

Any operation that touches the outside world (network, database, files) goes in an activity, never in the workflow body.

The SDK enforces this by default: a sandbox catches non-deterministic calls at write time with a clear error, rather than letting a subtle bug surface only on replay in production. This is a safety net, not a limitation. Frame it that way.

### Security and compliance questions

| Question they'll ask | Your answer |
|---|---|
| "Where does our data run?" | Your code and data run in your environment — your K8s, your VMs. Mistral hosts the orchestrator only. Workers connect outbound; Mistral never initiates connections into your network. |
| "Can we encrypt payloads?" | Yes. The SDK encrypts payloads before they leave the worker. Mistral stores ciphertext only. |
| "What about large payloads?" | Above 2 MB, the platform offloads to your blob storage (S3/GCS/Azure). Mistral stores only a reference. |
| "Can this run fully on-prem?" | Yes, for enterprise customers the full Studio environment can run in your private cloud or on-premises. |
| "Is there an audit trail?" | Every event in a workflow's lifecycle is recorded and queryable. OpenTelemetry traces work out of the box, no extra wiring. |
| "What's the underlying engine?" | Temporal — the same durable execution engine used at Netflix, Stripe, Salesforce, and JPMorgan Chase. Mistral extended it for AI-specific workloads (streaming, payload handling, multi-tenancy, observability). |

### What's not ready yet — be upfront about these

- **Pricing is not publicly listed.** Customers should expect to evaluate Studio access, model usage, worker deployment, and enterprise terms with Mistral directly. If a customer needs a contractual price commitment, escalate to [pricing owner — TBD]. Do not anchor with a number.

## Competitive positioning

> **Internal only. Do not share or screenshot for customers.**

### The objection you'll hear: "We can build this ourselves on Temporal."

This is the primary objection for an AI-orchestration product built on open infrastructure. It's a build-vs-buy conversation, not a feature comparison. Handle it in three moves:

**Acknowledge:** "You absolutely can. Temporal is open-source and battle-tested. That's why we built on it."

**Ask the trap question:** "Have you scoped the engineering effort to add AI-native streaming, payload encryption, multi-tenancy, human-in-the-loop with zero-compute pauses, OpenTelemetry observability, Le Chat distribution, and Studio's execution timeline on top of raw Temporal? What does the 12-month maintenance commitment look like for your platform team?"

**Pivot to value:** Workflows bundles the orchestration engine, the AI-specific extensions, the observability layer, and the end-user trigger surfaces (API + Studio + Le Chat) in one platform. The alternative is months of stitching — and then maintaining — Temporal + a streaming layer + an observability stack + an approval system + a distribution surface.

### Landscape positioning

| Alternative | Workflows advantage |
|---|---|
| **DIY Temporal** | No need to build the AI extensions, streaming, multi-tenancy, or observability yourself. No Temporal ops team required. |
| **LangChain / LangGraph** | Workflows provides durable execution. LangChain doesn't persist state across process restarts. If a node fails, you start over. |
| **AWS Bedrock AgentCore** | Cloud-agnostic. Runs on-prem. No AWS lock-in. European data sovereignty. |
| **Azure / Google agent tools** | Same cloud-agnosticism point. Mistral's open-weight models add model portability — customers aren't locked to one provider's models. |
